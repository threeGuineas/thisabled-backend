# 채팅 읽음 상태 설계

## 목표

수신자에게는 대화방별 및 전체 미읽음 메시지 수를 제공하고, 발신자에게는 수신자가 마지막으로 읽은 자신의 메시지 한 건만 읽음으로 표시한다.

## 범위와 용어

- 읽음 처리는 `GET /chat/rooms/{room_id}/messages`의 첫 페이지(`cursor` 없음)를 조회할 때 수행한다. 과거 페이지를 더 불러오는 조회는 읽음 위치를 바꾸지 않는다.
- 읽음 대상은 조회 사용자 본인이 보내지 않았고 `pending`이 아닌 메시지다. SAFE-03의 블러 메시지도 도착 사실은 보였으므로 읽음 대상으로 처리한다.
- 미읽음 수는 마지막 읽음 위치보다 뒤에 있는 읽음 대상 메시지의 개수다. 메시지 요청함도 방 단위 상태를 사용하므로 같은 규칙을 적용한다.
- 탈퇴로 `sender_id`가 `NULL`이 된 보존 메시지는 남은 참여자에게 수신 메시지로 취급한다.

## 대안과 결정

1. 메시지별 `read_at`은 개별 행을 대량 갱신해야 한다.
2. Redis 전용 상태는 재접속·다중 기기·영구 표시를 보장하지 못한다.
3. 채팅방·사용자별 마지막 읽음 위치를 저장하는 방식은 한 행만 갱신하고 필요한 두 UI를 모두 계산할 수 있다.

3번을 채택한다.

## 데이터 모델

`chat_read_states` 테이블을 추가한다.

| 열 | 규칙 |
| --- | --- |
| `room_id` | `chat_rooms.id` FK, 방 삭제 시 cascade |
| `user_id` | `users.id` FK, 사용자 삭제 시 cascade |
| `last_read_message_id` | 마지막으로 읽은 수신 메시지 UUID, 아직 읽은 메시지가 없으면 `NULL` |
| `last_read_at` | 마지막으로 읽은 수신 메시지의 생성 시각, 아직 없으면 `NULL` |

`(room_id, user_id)`는 유일하다. 읽음 상태 행은 첫 읽음 처리에서 upsert하므로 기존 방을 위한 데이터 마이그레이션은 필요 없다. 시간과 UUID를 함께 저장해 같은 시각의 메시지에도 목록 정렬과 동일한 위치 비교를 한다.

## API와 실시간 이벤트

- `GET /chat/rooms`와 `GET /chat/requests`의 각 방은 `unread_count`를 포함한다.
- `GET /chat/rooms`의 최상위 응답은 앱 배지용 `unread_total`을 포함한다.
- `GET /chat/rooms/{room_id}/messages`는 첫 페이지 조회 후 마지막 읽음 위치를 갱신한다. 각 `MessageOut`은 요청 사용자 본인이 보낸 메시지 중 상대방의 마지막 읽음 메시지와 일치할 때만 `is_read: true`다.
- 읽음 위치가 실제로 전진하면 상대 발신자에게 `chat.read` 이벤트를 보낸다. payload는 `room_id`와 `message_id`만 포함하며, 메시지 원문은 포함하지 않는다.

## 안전성과 일관성

- 방 참여자가 아닌 사용자는 기존과 같이 404를 받으며 읽음 상태를 갱신할 수 없다.
- 수신자가 실제로 볼 수 없는 `pending` 메시지는 읽음 위치와 미읽음 수에서 제외한다.
- 메시지 목록 응답을 만들기 전에 DB 트랜잭션을 커밋해 REST 재조회와 WebSocket 이벤트가 같은 읽음 위치를 보게 한다.
- 메시지 페이지네이션은 최신 메시지부터의 기존 정렬을 유지한다.

## 테스트

1. 수신자는 두 개의 새 메시지 후 방별 `unread_count: 2`와 `unread_total: 2`를 받는다.
2. 수신자의 첫 메시지 목록 조회는 미읽음 수를 0으로 만들고, 발신자 목록에서는 마지막 메시지 한 건만 `is_read: true`가 된다.
3. 발신자 자신의 메시지와 `pending` 메시지는 미읽음으로 세지 않는다.
4. 읽음 위치가 전진할 때만 `chat.read` WebSocket 이벤트가 발행되고 원문은 포함하지 않는다.
5. 기존 CHAT·SAFE·미디어·요청 흐름이 회귀하지 않는다.
